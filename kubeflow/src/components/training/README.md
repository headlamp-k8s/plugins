# Kubeflow Training Plugin — Operator & SRE Guide

A Headlamp plugin module that gives **platform operators, SREs, and ML engineers** deep Kubernetes-native observability over Kubeflow Training workloads, specifically focusing on the new unified Training Operator v2 architecture.

---

## What It Gives You

This plugin surfaces the underlying Kubernetes topology behind Kubeflow Training models. Instead of treating distributed training jobs as a black box, it exposes:

- **Worker & Launcher Pods:** Direct hierarchical access to child `JobSet`, `Job`, and `Pod` resources created by a `TrainJob`.
- **Resource Footprints:** Immediate visibility into GPU, CPU, and Memory usage requested by training nodes and pods.
- **Runtime Injection:** Clear visual diffs and JSON views of how a `TrainingRuntime` or `ClusterTrainingRuntime` is patched into a specific `TrainJob`.
- **Scheduling Integrity:** Insights into Volcano or Coscheduling configurations for distributed, multi-node workloads to ensure gang-scheduling is operating correctly.

## What It Offers

### Training Dashboard (`/kubeflow/training`)

A high-level overview page with:

- **Summary Cards:** Total TrainJobs, suspended workloads, and runtime definitions (Cluster vs. Namespace scoped).
- **Recent TrainJobs:** Quick access to the latest training runs and their execution phase.
- **Runtime Coverage:** A breakdown of available frameworks (e.g., PyTorch, MPI, XGBoost) and their configured scheduling policies.
- **Failed Workloads:** Instant triage of failed training jobs with surfacing of exact Kubernetes error reasons.

### TrainJobs (`/kubeflow/training/trainjobs`)

**List View:**

- Runtime reference, progress percentage, suspend state, and managed-by indicators.
- Child job readiness breakdown (e.g., `launcher:1/1, worker:2/2`).
- Row action: **View Primary Pod Logs** (automatically discovers the launcher/chief pod).

**Detail View:**

- **Workspace & Spec:** Complete visualization of the trainer image, arguments, environment variables, and initializers (dataset/model specs).
- **Runtime Resolution:** Preview the exact `TrainingRuntime` spec and the patched output generated by the TrainJob's `runtimePatches`.
- **Status & Metrics:** Progress percentage, estimated remaining time, and reported metrics (loss, accuracy, gpu_utilization).
- **Child Resources:** Hierarchical links to the generated `JobSet`, `Job`, and worker `Pod`s, including real-time phase and metric rollups.
- **Logs:** Instant access to primary pod logs via action buttons, or specific worker logs via the Logs tab.
- **Actions:** Suspend/Resume job action button directly in the UI.

### TrainingRuntimes & ClusterTrainingRuntimes (`/kubeflow/training/trainingruntimes`)

- Lists and detail views for both namespace-scoped and cluster-scoped runtimes.
- Displays `MLPolicy` details (e.g., Torch `numProcPerNode`, MPI implementations).
- Displays `PodGroupPolicy` for gang scheduling (Volcano queues, Coscheduling timeouts).
- Lists all `TrainJobs` that reference the current runtime.
- **Warnings:** Flags if a runtime is missing framework labels or gang scheduling setups.

## Data Integrity

All data flows through the standard Kubernetes API server to the unified Trainer v2 API:

```text
Headlamp Backend → Kubernetes API Server → CRD API Groups
  /apis/trainer.kubeflow.org/v1alpha1/trainjobs
  /apis/trainer.kubeflow.org/v1alpha1/trainingruntimes
  /apis/trainer.kubeflow.org/v1alpha1/clustertrainingruntimes
```

The `SectionPage` wrapper performs group-level discovery. If the `trainer.kubeflow.org` group is missing, the UI provides a clear guided error message instead of an empty screen or hanging.

## Developer Testing Guide

### Prerequisites

- [kind](https://kind.sigs.k8s.io/) or [minikube](https://minikube.sigs.k8s.io/)
- `kubectl` configured
- Node.js 18+

### Step 1: Create a Local Cluster

```bash
kind create cluster --name headlamp-kubeflow
kubectl config use-context kind-headlamp-kubeflow
```

### Step 2: Install CRDs

Apply the Trainer v2 CRDs (no controllers needed for UI development):

```bash
kubectl apply -f https://raw.githubusercontent.com/kubeflow/manifests/master/applications/trainer/upstream/base/crds/trainer.kubeflow.org_trainjobs.yaml
kubectl apply -f https://raw.githubusercontent.com/kubeflow/manifests/master/applications/trainer/upstream/base/crds/trainer.kubeflow.org_trainingruntimes.yaml
kubectl apply -f https://raw.githubusercontent.com/kubeflow/manifests/master/applications/trainer/upstream/base/crds/trainer.kubeflow.org_clustertrainingruntimes.yaml
```

### Step 3: Create Test Data

Create a namespace and apply sample training resources:

```bash
kubectl create namespace kubeflow-user

# Apply a local mock of a TrainingRuntime
kubectl apply -f test_files/trainer/trainingruntime.yaml

# Apply a local mock of a ClusterTrainingRuntime
kubectl apply -f test_files/trainer/clustertrainingruntime.yaml

# Apply a local mock of a TrainJob
kubectl apply -f test_files/trainer/trainjob-running.yaml
```

### Step 4: Run the Plugin

```bash
cd plugins/kubeflow
npm install
npm run start     # Watches for changes, auto-deploys to ~/.config/Headlamp/plugins/
```

### Step 5: Verify

1. Open Headlamp and select your local cluster.
2. Navigate to **Kubeflow** → **Training**.
3. Verify the Dashboard summary cards reflect the test data.
4. Navigate to **TrainJobs**, **TrainingRuntimes**, and **ClusterTrainingRuntimes** to verify detail views and tab outputs.
5. Check that the "View Primary Pod Logs" and "Suspend" actions render successfully.

## Running Tests

### Unit Tests

```bash
cd plugins/kubeflow
npm test
```

Tests cover utility functions in `trainerUtils.ts`:

- Parsing CPU and Memory constraints.
- Correctly resolving namespace-scoped vs. cluster-scoped runtimes.
- Checking runtime patch diffing accuracy.

### Storybook

```bash
cd plugins/kubeflow
npm run storybook
```

Storybook stories isolate the presentational components using realistic mock data, including active metrics and child job references. Available stories:

- `Kubeflow/Training/TrainingOverview` — Complete dashboard view
- `Kubeflow/Training/TrainJobsList` — Resource list view
- `Kubeflow/Training/TrainJobsDetail` — Deep inspection view with diffs
- `Kubeflow/Training/TrainingRuntimeDetails` — Runtime visualization view
